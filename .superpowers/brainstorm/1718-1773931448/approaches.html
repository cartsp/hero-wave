Implementation Approaches

Three ways to build the WavyBackground Blazor component — all use Canvas + JS interop under the hood

A

Single-Project Component (Recommended)

Build the component directly inside a Blazor WASM app. The WavyBackground.razor component, its JS module, and a demo page all live in one project.

Pros

  • Simplest setup — one project, fast iteration
  • Easy to test and demo immediately
  • Can extract into a library later if needed

Cons

  • Component isn't separately packaged for reuse across solutions
B

Razor Class Library + Demo App

Two projects in a solution: a Razor Class Library (RCL) containing the component + bundled JS, and a Blazor WASM app that references it for demo/testing.

Pros

  • Component is a NuGet-packageable library from day one
  • Clean separation of component vs consumer
  • JS files auto-bundled via _content/ static assets

Cons

  • Slightly more setup (solution + two projects)
C

Standalone JS + Blazor Wrapper

Build the wave engine as a standalone ES module first (testable in plain HTML), then create a thin Blazor component wrapper that calls into it.

Pros

  • Wave engine is framework-agnostic, reusable beyond Blazor
  • Can debug the JS visuals independently in a browser

Cons

  • Two separate development workflows (JS + Blazor)
  • More moving parts for a Blazor-focused use case
